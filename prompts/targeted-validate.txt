- **Document Everything:** Even if Shannon finds zero new vulnerabilities, that is a result that must be documented. The absence of findings is as important as the presence of findings.
</critical>

<methodology>

## Shannon Validation Process

### 1) Pre-Validation Checks
- Verify the remediation branch is checked out: `git branch --show-current`
- Verify all fix agent commits are present: `git log --oneline`
- Check for uncommitted changes: `git status`
- If there are uncommitted changes, either commit them or stash them
- Read `deliverables/fix_review_report.md` to check for any unresolved issues that might affect validation

### 2) Prepare the Validation Environment
- Create the validation output directory: `mkdir -p deliverables/validation`
- Ensure the target application is running with the patched codebase
- If the application needs to be restarted to pick up code changes, restart it
- Verify the application is accessible at the target URL

### 3) Execute the Shannon Pipeline
- Run Shannon against the patched codebase with output directed to `deliverables/validation/`
- Shannon execution command (adapt based on the Shannon installation):
  ```
  ./shannon start URL=<target_url> REPO=<repo_path> OUTPUT=./deliverables/validation/
  ```
- Monitor the pipeline progress through Shannon's logs
- Create TodoWrite tasks for each Shannon phase:
  - Pre-Recon phase
  - Recon phase
  - Vulnerability Analysis phase (5 parallel agents)
  - Exploitation phase (5 parallel agents)
  - Reporting phase

### 4) Monitor Pipeline Progress
- Check Shannon's workflow status periodically
- If a phase fails, document the failure and attempt to restart
- Shannon's five-phase pipeline must complete fully:
  1. Pre-Recon: External scans + source code analysis
  2. Recon: Attack surface mapping from initial findings
  3. Vulnerability Analysis: 5 parallel agents (injection, xss, auth, authz, ssrf)
  4. Exploitation: 5 parallel agents validating vulnerabilities
  5. Reporting: Executive-level security report

### 5) Collect Validation Results
Once Shannon completes, verify the following files exist in `deliverables/validation/`:
- `comprehensive_security_assessment_report.md` - The validation pentest report
- Exploitation queue JSONs (may be empty if vulnerabilities are fixed)
- Exploitation evidence files (may be empty)
- Analysis deliverables

### 6) Create the Validation Summary
Read Shannon's validation output and create a high-level summary:
- Count of findings by category (injection, xss, auth, ssrf, authz)
- Count of EXPLOITED vs POTENTIAL vs FALSE_POSITIVE findings
- Compare at a high level with the original findings count
- Note any NEW findings that did not exist in the original report (potential regressions)

### 7) Handle Edge Cases

**If Shannon cannot run:**
- Document why (application not accessible, Shannon not installed, configuration issue)
- Provide as much manual validation as possible (re-test witness payloads manually)
- Flag this as a blocking issue in the validation summary

**If Shannon finds the same vulnerabilities:**
- This means the fixes did not work. Document which fix groups failed.
- Cross-reference with the fix review report to understand why

**If Shannon finds new vulnerabilities:**
- These are potential regressions introduced by the fixes
- Document them prominently in the validation summary
- The Compare agent will classify them as REGRESSION

**If Shannon finds zero vulnerabilities:**
- This is the ideal outcome. Document it.
- Still capture all Shannon output files for the Compare agent

</methodology>
