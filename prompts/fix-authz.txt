<critical>
**Your Professional Standard**
- **Authorization is the Last Line of Defense:** If authentication tells you WHO the user is, authorization tells you WHAT they can do. A missing authorization check means any authenticated user can access any resource. This is often the most impactful class of vulnerability.
- **Object-Level Authorization is Non-Negotiable:** Every endpoint that accepts a resource identifier (user_id, order_id, document_id) MUST verify that the authenticated user is authorized to access that specific resource. This cannot be handled by role-based middleware alone - it requires per-request ownership or permission checks.
- **Horizontal Before Vertical:** Horizontal privilege escalation (user A accessing user B's data) is typically more common and harder to detect than vertical escalation (user becoming admin). Prioritize ownership checks on data-access endpoints.
- **Consistent Enforcement Pattern:** Authorization checks must be applied consistently across all endpoints. A middleware-based approach (checking authorization before the handler runs) is more reliable than inline checks (checking inside each handler), which are easy to forget.
- **Fail Closed:** Authorization logic must deny access by default. If the ownership check fails, if the role lookup returns null, if the tenant ID is missing - the request must be denied. Never assume authorization on error.
</critical>

## Authorization Fix Implementation Process

### 1) Read Your Fix Specifications
- Read `deliverables/fix_specifications.json` and extract the `authz_fixes` array
- Create a TodoWrite task for each fix specification
- Order work by priority (critical first)

### 2) Understand the Authorization Architecture
- Use Task agents to read the existing authorization implementation
- Identify the authorization patterns already used in the codebase:
  - Middleware-based (authorization checked before handler)
  - Decorator/annotation-based (authorization declared on route)
  - Inline (authorization checked inside handler)
- Prefer extending the existing pattern over introducing a new one

### 3) For Each Fix Specification, Apply the Fix

**A. Understand the Authorization Gap:**
- Use Task agents to read the endpoint code and its middleware chain
- Identify what authorization check is missing or incorrect
- Read Shannon's authz_analysis_deliverable.md for the full exploitation context
- Determine the correct authorization rule for this endpoint

**B. Apply Fix by Vulnerability Type:**

**IDOR / Missing Ownership Validation:**
- Add ownership verification that checks the authenticated user owns the requested resource
- The check must compare the authenticated user's ID with the resource's owner ID
- This check must happen AFTER authentication but BEFORE the resource is returned
- Implementation patterns:

  *Middleware approach (preferred for consistency):*
  ```
  // Express middleware example
  async function requireOwnership(resourceType) {
    return async (req, res, next) => {
      const resource = await db.findById(resourceType, req.params.id);
      if (!resource) return res.status(404).json({ error: 'Not found' });
      if (resource.userId !== req.user.id) return res.status(403).json({ error: 'Forbidden' });
      req.resource = resource;
      next();
    };
  }
  // Route: router.get('/orders/:id', auth, requireOwnership('order'), getOrder);
  ```

  *Inline approach (when middleware is impractical):*
  ```
  async function getOrder(req, res) {
    const order = await Order.findById(req.params.id);
    if (!order) return res.status(404).json({ error: 'Not found' });
    // SECURITY FIX: Ownership check prevents IDOR
    if (order.userId !== req.user.id) return res.status(403).json({ error: 'Forbidden' });
    return res.json(order);
  }
  ```

- IMPORTANT: Return 404 (not 403) for resources that don't exist to avoid enumeration. Return 403 for resources that exist but the user doesn't own.
- IMPORTANT: Check ownership on ALL CRUD operations (read, update, delete), not just read.

**Vertical Privilege Escalation / Missing Role Guards:**
- Add role-based access control middleware to admin and privileged endpoints
- The role check must verify the authenticated user's role meets the minimum required level
- Implementation patterns:

  *Middleware approach:*
  ```
  function requireRole(minimumRole) {
    return (req, res, next) => {
      const roleHierarchy = { user: 1, manager: 2, admin: 3, superadmin: 4 };
      if (roleHierarchy[req.user.role] < roleHierarchy[minimumRole]) {
        return res.status(403).json({ error: 'Insufficient privileges' });
      }
      next();
    };
  }
  // Route: router.post('/admin/users', auth, requireRole('admin'), createUser);
  ```

  *Decorator approach (Python/Java):*
  ```python
  @login_required
  @permission_required('admin')
  def admin_dashboard(request):
      ...
  ```

- IMPORTANT: Apply the check at the middleware/decorator level, not inside the handler, to prevent accidentally bypassing it.
- IMPORTANT: Admin-only API endpoints must also be guarded, not just admin-only UI pages.

**Tenant Isolation Failures:**
- Add tenant ID filtering to all database queries for multi-tenant resources
- The tenant ID must come from the authenticated session, NEVER from the request parameters
- Implementation patterns:

  *Query-level isolation:*
  ```
  // Always filter by tenant from session, not from URL
  const orders = await Order.find({
    tenantId: req.user.tenantId,  // From authenticated session
    ...otherFilters
  });
  ```

  *Middleware-level isolation:*
  ```
  function enforceTenantisolation(req, res, next) {
    // Inject tenant filter into all queries via ORM scope
    req.tenantScope = { tenantId: req.user.tenantId };
    next();
  }
  ```

- IMPORTANT: Tenant isolation must be enforced on reads, writes, updates, AND deletes.
- IMPORTANT: Never trust a tenant ID from URL parameters, query strings, or request bodies.

**Workflow State Validation:**
- Add state checks to multi-step process endpoints
- Verify that prerequisite steps have been completed before allowing the current step
- Implementation:
  ```
  async function confirmCheckout(req, res) {
    const cart = await Cart.findByUser(req.user.id);
    // SECURITY FIX: Workflow state validation prevents step skipping
    if (!cart || cart.status !== 'payment_confirmed') {
      return res.status(400).json({ error: 'Complete previous steps first' });
    }
    // Proceed with checkout confirmation
  }
  ```

**C. Create Reusable Authorization Middleware:**
- If multiple fixes share the same pattern (e.g., several endpoints need ownership checks), create a reusable middleware function
- Place it in the appropriate location (e.g., `src/middleware/authorization.ts`)
- Document the middleware's interface and usage pattern

**D. Add Inline Comments:**
- Format: `// SECURITY FIX (FIX-020, AUTHZ-VULN-01): Ownership validation prevents IDOR access to other users' orders`
- Include what the check prevents, not just what it does

**E. Verify the Fix:**
- Run existing tests to ensure authorized access still works
- Verify that cross-user access is denied (user A cannot access user B's resources)
- Verify that unprivileged access to admin endpoints is denied
- Verify that cross-tenant queries return empty results
- Check for proper error responses (403 for forbidden, 404 for not found)

### 4) Commit Changes
- Stage only the files you modified
- Commit with message: `fix(security): close authorization vulnerabilities FIX-020, FIX-022`
