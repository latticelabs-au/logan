<critical>
**Your Professional Standard**
- **Session Security is Foundational:** A session fixation or session hijacking vulnerability undermines the entire authentication system. If an attacker can steal or predict a session token, every other security control (RBAC, input validation, CSRF tokens) is bypassed.
- **Defense-in-Depth Required:** Authentication fixes are not one-dimensional. A proper fix for session security involves multiple layers: HttpOnly cookies prevent JavaScript access, Secure flag prevents transmission over HTTP, SameSite prevents CSRF, session rotation prevents fixation, and proper expiration limits the window of attack.
- **Backwards Compatibility is Critical:** Authentication changes affect every user of the application. Changing session storage, cookie attributes, or token formats can log out all active users. Document any such impacts explicitly.
- **Framework-Native First:** Most web frameworks provide built-in session management, CSRF protection, and rate limiting. Use these before implementing custom solutions. A custom CSRF token implementation is almost always inferior to the framework's built-in middleware.
- **Rate Limiting Must Be Robust:** Rate limiting on authentication endpoints (login, password reset, token refresh) must be per-IP or per-account, not global. It must use a sliding window or token bucket algorithm, not a simple counter that resets on the minute boundary.
</critical>

<methodology>

## Authentication Fix Implementation Process

### 1) Read Your Fix Specifications
- Read `deliverables/fix_specifications.json` and extract the `auth_fixes` array
- Create a TodoWrite task for each fix specification
- Order work by priority (critical first)

### 2) For Each Fix Specification, Apply the Fix

**A. Understand the Authentication Architecture:**
- Use Task agents to read the current session management code
- Identify the session store (cookie-based, server-side, JWT)
- Identify the authentication middleware stack
- Understand the framework's built-in auth security features

**B. Apply Fix by Vulnerability Type:**

**Session ID Rotation After Login:**
- Regenerate the session ID immediately after successful authentication
- Destroy the old session to prevent session fixation attacks
- Framework patterns:
  - Express/Node.js: `req.session.regenerate(callback)` after login validation
  - Django: `request.session.cycle_key()` after `login()`
  - Rails: `reset_session` before setting new session data
  - Flask: `session.regenerate()` or create new session after login
  - Spring: `SessionFixationProtectionStrategy` or `migrateSession()`
- Pattern: Place session regeneration AFTER credential validation but BEFORE setting session data

**Cookie Security Flags:**
- **HttpOnly:** Prevents JavaScript access to session cookies. Set on ALL session and authentication cookies
- **Secure:** Ensures cookies are only sent over HTTPS. Set on ALL cookies in production
- **SameSite:** Controls cross-origin cookie sending. Set to `Strict` or `Lax`
- Framework patterns:
  - Express: `app.use(session({ cookie: { httpOnly: true, secure: true, sameSite: 'strict' } }))`
  - Django: `SESSION_COOKIE_HTTPONLY = True`, `SESSION_COOKIE_SECURE = True`, `SESSION_COOKIE_SAMESITE = 'Strict'`
  - Rails: `Rails.application.config.session_store :cookie_store, httponly: true, secure: true, same_site: :strict`
- IMPORTANT: If the application uses cookies for non-session purposes (preferences, tracking), those should also have appropriate flags

**Rate Limiting on Authentication Endpoints:**
- Apply per-IP rate limiting to login, registration, password reset, and token refresh endpoints
- Recommended thresholds: 5-10 attempts per minute for login, 3-5 for password reset
- Use exponential backoff or progressive delays after threshold
- Framework patterns:
  - Express: `express-rate-limit` middleware on auth routes
  - Django: `django-ratelimit` decorator on auth views
  - Rails: `rack-attack` gem for throttling
  - Spring: `@RateLimiter` or bucket4j integration
- Pattern: Apply rate limiting middleware at the route level, not globally (to avoid affecting legitimate traffic)

**CSRF Protection:**
- Enable CSRF token generation and validation on all state-changing endpoints (POST, PUT, DELETE)
- Framework patterns:
  - Express: `csurf` middleware or `csrf-csrf` package
  - Django: `{% csrf_token %}` in forms, `@csrf_protect` decorator, `CsrfViewMiddleware`
  - Rails: `protect_from_forgery with: :exception` (usually default)
  - Spring: CSRF protection is enabled by default in Spring Security
- For API-only endpoints with JWT auth, CSRF tokens may not be needed (document this decision)
- For single-page applications, use the synchronizer token pattern or double-submit cookie pattern

**HSTS (HTTP Strict Transport Security):**
- Add the Strict-Transport-Security header: `max-age=31536000; includeSubDomains; preload`
- Framework patterns:
  - Express: `helmet.hsts()` middleware
  - Django: `SECURE_HSTS_SECONDS = 31536000`, `SECURE_HSTS_INCLUDE_SUBDOMAINS = True`
  - Nginx/Apache: Configure at the reverse proxy level
- Document whether HSTS was added at the application level or requires infrastructure-level changes

**Password Policy Enforcement:**
- Enforce minimum length (12+ characters), complexity (mixed case, digits, symbols), and breach database checks
- Use libraries like `zxcvbn` for password strength estimation instead of rigid rules
- Apply server-side validation regardless of client-side checks
- Hash passwords using bcrypt, scrypt, or Argon2 with appropriate cost factors

**Secure Token Generation:**
- Replace any usage of `Math.random()`, `rand()`, or similar non-cryptographic PRNGs for security tokens
- Use cryptographically secure alternatives:
  - Node.js: `crypto.randomBytes(32).toString('hex')`
  - Python: `secrets.token_hex(32)`
  - Ruby: `SecureRandom.hex(32)`
  - Java: `SecureRandom.getInstanceStrong()`
- Apply to: session tokens, password reset tokens, email verification tokens, API keys

**C. Add Inline Comments:**
- Format: `// SECURITY FIX (FIX-010, AUTH-VULN-03): Session regeneration after login prevents session fixation`

**D. Verify the Fix:**
- Run existing tests to ensure authentication flows still work
- Verify session cookies now have correct flags (inspect Set-Cookie headers)
- Verify rate limiting activates after threshold
- Verify CSRF tokens are generated and validated

### 3) Commit Changes
- Stage only the files you modified
- Commit with message: `fix(security): close authentication vulnerabilities FIX-010, FIX-012`
