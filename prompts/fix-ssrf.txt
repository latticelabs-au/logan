<critical>
**Your Professional Standard**
- **URL Parsing is Treacherous:** URL parsers disagree on edge cases. An attacker can craft URLs that pass your validation but resolve to internal addresses when fetched. Always resolve the URL to an IP address and validate the IP, not just the hostname string.
- **DNS Rebinding is Real:** An attacker can configure a DNS record that first resolves to a public IP (passing validation) and then resolves to 127.0.0.1 (reaching internal services). Your fix must validate the resolved IP at connection time, not just at URL parse time. If the HTTP client supports a connect callback or socket pre-connect hook, use it.
- **Redirects Are an Attack Vector:** Even if you validate the initial URL, the server may redirect to an internal address. Either disable redirect following entirely, or validate each redirect target against the same rules as the initial URL.
- **Allowlisting Over Denylisting:** When possible, use an allowlist of permitted domains/IP ranges rather than a denylist of private ranges. An allowlist is more robust because it protects against unknown private ranges, IPv6 mapped addresses, and other edge cases.
- **Cloud Metadata is a High-Value Target:** The most impactful SSRF targets are cloud provider metadata endpoints (169.254.169.254 for AWS/GCP, 169.254.169.254 for Azure). Blocking the entire 169.254.0.0/16 range is essential.
</critical>

<methodology>

## SSRF Fix Implementation Process

### 1) Read Your Fix Specifications
- Read `deliverables/fix_specifications.json` and extract the `ssrf_fixes` array
- Create a TodoWrite task for each fix specification
- Order work by priority (critical first)

### 2) For Each Fix Specification, Apply the Fix

**A. Understand the Outbound Request Pattern:**
- Use Task agents to read the current code at the specified file and line range
- Identify the HTTP client being used (axios, fetch, requests, http.Client, etc.)
- Identify where user input enters the URL construction
- Understand what legitimate URLs the feature needs to fetch

**B. Apply Fix by Defense Layer:**

**Layer 1: URL Scheme Restriction**
- Restrict the URL scheme to `https` only (or `http` + `https` if the feature requires HTTP)
- Reject `file://`, `gopher://`, `dict://`, `ftp://`, `ldap://`, and all other schemes
- Implementation:
  ```
  const parsedUrl = new URL(userInput);
  if (!['http:', 'https:'].includes(parsedUrl.protocol)) {
    throw new Error('Invalid URL scheme');
  }
  ```
- Apply BEFORE any DNS resolution or HTTP request

**Layer 2: URL Allowlisting (Preferred) or Private IP Denylisting**

*Option A - Domain Allowlist (strongest, use when possible):*
- Maintain an explicit list of allowed external domains
- Validate the parsed URL's hostname against the allowlist
- Implementation:
  ```
  const ALLOWED_DOMAINS = ['api.example.com', 'cdn.example.com'];
  if (!ALLOWED_DOMAINS.includes(parsedUrl.hostname)) {
    throw new Error('Domain not in allowlist');
  }
  ```

*Option B - Private IP Range Blocking (use when allowlisting is impractical):*
- Resolve the hostname to an IP address BEFORE making the request
- Validate the resolved IP against blocked ranges
- Blocked ranges (IPv4):
  - `127.0.0.0/8` (loopback)
  - `10.0.0.0/8` (private)
  - `172.16.0.0/12` (private)
  - `192.168.0.0/16` (private)
  - `169.254.0.0/16` (link-local, cloud metadata)
  - `0.0.0.0/8` (unspecified)
  - `100.64.0.0/10` (carrier-grade NAT)
  - `198.18.0.0/15` (benchmark testing)
- Blocked ranges (IPv6):
  - `::1/128` (loopback)
  - `fd00::/8` (unique local)
  - `fe80::/10` (link-local)
  - `::ffff:0:0/96` (IPv4-mapped, check the mapped address against IPv4 rules)
- Implementation patterns:
  - Node.js: Use `dns.resolve()` to get the IP, then check against blocked ranges using `ipaddr.js` or manual CIDR matching
  - Python: Use `socket.getaddrinfo()` then check with `ipaddress.ip_address()` against `ip_network()` ranges
  - Go: Use `net.LookupIP()` then check against `net.ParseCIDR()` ranges

**Layer 3: Redirect Validation**

*Option A - Disable redirect following (simplest):*
- Configure the HTTP client to not follow redirects
- Node.js axios: `{ maxRedirects: 0 }` or `{ followRedirect: false }`
- Python requests: `allow_redirects=False`
- Go: Set `CheckRedirect` to return an error
- Then handle the redirect response manually if needed

*Option B - Validate each redirect target:*
- Hook into the redirect handler to validate each redirect URL against the same rules
- Node.js axios: Use `beforeRedirect` callback to validate
- Python requests: Use `Session` with custom `rebuild_auth` and URL validation in redirect hooks
- Apply the same scheme restriction and IP validation to each redirect target

**Layer 4: DNS Rebinding Protection**

*Approach A - Pin DNS resolution:*
- Resolve DNS once, validate the IP, then connect directly to the validated IP
- Set the Host header to the original hostname
- This prevents TOCTOU attacks where DNS changes between validation and connection

*Approach B - Connection-time validation:*
- If the HTTP client supports a socket pre-connect callback, validate the resolved IP at connection time
- Node.js: Use `agent` with custom `createConnection` that validates the remote IP
- This is the strongest defense against DNS rebinding

**C. Create a Reusable Validation Function:**
- If multiple SSRF fixes target different files but share the same validation logic, create a shared utility function
- Place it in the appropriate location (e.g., `src/utils/url-validator.ts` or `src/lib/safe_fetch.py`)
- Have all SSRF-affected code paths use this shared function
- Pattern:
  ```
  async function safeFetch(url: string, options?: RequestInit): Promise<Response> {
    // 1. Parse and validate scheme
    // 2. Resolve DNS and validate IP
    // 3. Make request with redirect validation
  }
  ```

**D. Add Inline Comments:**
- Format: `// SECURITY FIX (FIX-015, SSRF-VULN-01): URL validation prevents server-side request forgery`

**E. Verify the Fix:**
- Run existing tests to ensure outbound request features still work for legitimate URLs
- Verify that internal IPs (127.0.0.1, 10.0.0.1, 169.254.169.254) are rejected
- Verify that non-http schemes are rejected
- Verify that redirects to internal IPs are blocked (if redirect following is enabled)

### 3) Commit Changes
- Stage only the files you modified
- Commit with message: `fix(security): close SSRF vulnerabilities FIX-015, FIX-017`
