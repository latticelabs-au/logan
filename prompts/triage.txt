<methodology>

## Systematic Triage Process

### 1) Inventory All Shannon Deliverables
- Use the Read tool to check which deliverable files exist in the deliverables/ directory
- Create a TodoWrite task for each existing deliverable that needs processing
- Note which vulnerability types had findings and which had none

### 2) Parse Exploitation Evidence First (Highest Signal)
- Read each exploitation evidence file (injection, xss, auth, ssrf, authz)
- Extract every vulnerability with verdict EXPLOITED - these are your highest-priority items
- Extract every vulnerability with verdict POTENTIAL or BLOCKED_BY_SECURITY
- Record: vuln ID, type, endpoint, severity, affected files, exploitation method

### 3) Cross-Reference with Analysis Deliverables
- For each EXPLOITED vulnerability, read the corresponding analysis deliverable to get the full source-to-sink trace
- The analysis deliverable contains the root cause information (e.g., which function lacks sanitization, which middleware is missing)
- Record: source file:line, sink file:line, sanitization gaps, code patterns

### 4) Cross-Reference with Exploitation Queues
- Parse each JSON exploitation queue to ensure no vulnerability was missed
- The queues contain structured data (vuln ID, type, source, sink, confidence) that is easier to process than prose reports
- Reconcile: every queue entry should map to an evidence entry. Flag any gaps.

### 5) Read Architectural Context
- Read pre_recon_deliverable.md to understand the codebase structure, frameworks, and security patterns
- This informs your root cause grouping: if the codebase uses a shared input validation layer, multiple bypasses of that layer share a root cause

### 6) Classify Every Finding
Apply these classification rules strictly:
- **EXPLOITED:** Shannon's exploitation agent successfully demonstrated impact (data extracted, unauthorized access, code execution). These MUST be fixed.
- **POTENTIAL:** Shannon's analysis identified a vulnerable code path but exploitation was blocked by external factors (not security controls). These SHOULD be fixed.
- **INFORMATIONAL:** Findings that are purely theoretical, require internal access, or were confirmed as false positives by exploitation. These are SKIPPED with documented justification.

### 7) Group by Root Cause
This is your most critical analytical step. For each pair of vulnerabilities, ask:
- Do they share the same unsanitized input path?
- Do they share the same missing middleware or security check?
- Would fixing one automatically fix the other?
- Are they in the same file or module?

If YES to any of these, they belong in the same fix group.

**Root Cause Patterns to Look For:**
- Multiple SQLi from the same raw query builder function
- Multiple XSS from the same template rendering path without encoding
- Multiple auth issues from the same session management implementation
- Multiple SSRF from the same URL fetcher without validation
- Multiple authz issues from the same missing ownership check pattern

### 8) Assign Priority and Order
- **CRITICAL:** EXPLOITED vulnerabilities with data extraction or RCE demonstrated
- **HIGH:** EXPLOITED vulnerabilities with auth bypass or privilege escalation
- **MEDIUM:** POTENTIAL vulnerabilities with clear code path but unconfirmed exploitation
- **LOW:** POTENTIAL vulnerabilities with uncertain code path or low-impact

Within each priority level, order by:
1. Number of endpoints affected (wider blast radius first)
2. Sensitivity of data at risk
3. Ease of exploitation (lower barrier = higher priority)

### 9) Determine Fix Strategy Per Group
For each fix group, specify the remediation approach:
- **Injection:** parameterized_query, prepared_statement, input_validation, command_array, path_canonicalization, safe_deserialization
- **XSS:** output_encoding, csp_header, template_autoescape, dom_purify, attribute_encoding
- **Auth:** session_rotation, cookie_flags, rate_limiting, csrf_token, password_policy, secure_token_generation
- **SSRF:** url_allowlist, private_ip_blocking, protocol_restriction, redirect_validation, dns_rebinding_protection
- **Authz:** ownership_middleware, rbac_guard, tenant_isolation, object_authorization, workflow_validation

### 10) Produce the Remediation Plan
Generate `deliverables/remediation_plan.json` following the exact schema specified in the deliverable instructions.

**Field Requirements:**
- `id`: Sequential FIX-NNN identifiers starting at FIX-001
- `priority`: Based on exploitation evidence and impact

- `root_cause`: Plain-language description of the shared flaw
- `root_cause_location`: The primary file:line where the fix should be applied
- `affected_files`: ALL files that need modification for this fix group
- `fix_strategy`: The remediation technique to apply
- `dependencies`: Array of other FIX-NNN IDs that must be completed first (empty if independent)
- `estimated_complexity`: trivial (config change), simple (single function), moderate (multiple files), complex (architectural change)
