<critical>
**Your Professional Standard**
- **Blind Audit:** This is a FRESH penetration test, not a re-check. Shannon must approach the codebase with no preconceptions. Do NOT feed it the original vulnerability list, fix locations, or any prior findings. Let Shannon discover whatever it discovers.
- **Full Pipeline, No Shortcuts:** Every phase must run. Pre-recon code analysis, live recon with browser automation, all 5 parallel vulnerability analysis agents, conditional exploitation, and reporting. Skipping any phase invalidates the audit.
- **Isolated Output:** All Shannon output goes to `deliverables/validation/full-audit/` to keep it separate from both the original findings and the targeted validation results.
- **This Catches What Targeted Validation Cannot:** The targeted validation (previous phase) confirmed the 20 known vulns are fixed. This phase catches:
  - Regressions: fixes that introduced NEW vulnerabilities
  - Incomplete fixes: vulns that appear fixed for the known payload but are exploitable via different vectors
  - Blind spots: attack surfaces Shannon didn't find in the original audit but discovers now
  - Fix side effects: new code paths introduced by the remediation that create new attack surfaces
- **Patience:** A full Shannon audit takes 60-90 minutes. Do not interrupt it. The Compare agent needs this complete output to produce the final before/after analysis.
</critical>

<methodology>

## Full Audit Execution

### 1) Prepare the Audit Environment
- Verify the remediation branch is checked out and clean (`git status`)
- Create output directory: `mkdir -p deliverables/validation/full-audit`
- Verify the target application is reachable

### 2) Invoke Shannon
- Run the full Shannon pipeline against the current repository state
- Shannon must run ALL phases: pre-recon, recon, vulnerability analysis (5 agents), exploitation (conditional), reporting
- Direct Shannon's output to the validation subdirectory
- Do NOT provide Shannon with any prior findings or fix information — this must be a blind audit

### 3) Monitor and Capture Output
- Wait for Shannon to complete all phases
- Capture the full Shannon report and all intermediate deliverables
- If Shannon fails mid-pipeline, document what completed and what didn't

### 4) Analyze Results
- Read Shannon's output report
- Categorize any findings:
  - **REGRESSION:** A vulnerability that exists in the patched code but did NOT exist in the original code (introduced by a fix)
  - **PERSISTENT:** A vulnerability that Shannon found in both the original and patched code (fix was ineffective)
  - **NEW_DISCOVERY:** A vulnerability Shannon found now but missed in the original audit (different attack path or expanded coverage)
  - **CLEAN:** No findings — all previously-found vulns are confirmed fixed and no new ones discovered

### 5) Generate Summary Report
Structure your report as:

```markdown
# Shannon Full Audit Report (Post-Remediation)

## Audit Metadata
- Audit date: {date}
- Shannon pipeline phases completed: {list}
- Total findings: {count}
- Regressions: {count}
- New discoveries: {count}

## Findings
{For each finding, classify as REGRESSION / PERSISTENT / NEW_DISCOVERY with full details}

## Clean Bill of Health
{If no findings, explicitly state this with evidence from Shannon's report}

## Shannon Pipeline Output
{Summary of what each Shannon phase found, with references to full deliverables in validation/full-audit/}
```
